## Список логов:

### Уровни логирования

#### Политика уровней логирования

* DEBUG - используется только для локальной отладки и временного расследования инцидентов; в production постоянно не включается.
* INFO - используется для штатных бизнес-событий и ожидаемых действий системы, которые важны для восстановления хода обработки заказа.
* WARN - используется для нештатных, но обратимых ситуаций, при которых система продолжает работу, но требуется внимание команды.
* ERROR - используется для ошибок, из-за которых операция не была выполнена, выполнена некорректно или требуется немедленная реакция.

#### Предполагается использование следующих уровней логирования:

DEBUG:

* временно при расследовании инцидентов
* не используется постоянно в production

INFO:

* штатные бизнес-события

WARN:

* retry сообщений
* задержки обработки
* частичные ошибки

ERROR:

* ошибки обработки заказа
* отправка в DLQ
* падение интеграции

#### Поля:

* timestamp
* service_name
* environment
* order_id
* customer_id или client_id
* user_id / api_partner_id
* status_from
* status_to
* request_id
* trace_id
* message_id если есть очередь
* result=success/fail

### Shop API

#### INFO:

* создание заказа
* загрузка 3D-файла
* отправка заказа в обработку
* изменение статуса заказа
* вызов внешнего API / публикация сообщения в RabbitMQ
* успешный ответ клиенту по заказу

### CRM API

#### INFO:

* получение заказа из очереди
* создание/обновление заказа в CRM
* подтверждение производства
* закрытие заказа
* ручные действия продавца по заказу

### MES API

#### INFO:

* получение заказа на расчёт
* завершение расчёта стоимости
* взятие заказа оператором в работу
* смена производственных статусов
* завершение / упаковка / отправка

### RabbitMQ-интеграция

#### INFO:

* публикация сообщения
* получение сообщения consumer’ом
* успешный ack
* retry
* отправка в DLQ

## Мотивация:

В текущем состоянии информация об инцидентах поступает от клиентов. Также сейчас нет единой картины событий внутри сервисов. Это увеличивает MTTR и
увеличивает нагрузку на поддержку и увеличивает время поиска причин ошибок. Централизованное логирование позволит быстро находить события по
`order_id`, `trace_id`, `message_id` и `customer_id`. Логи должны коррелироваться между сервисами по `trace_id` и `order_id` для построения сквозной
картины обработки заказа. Наличие логов уменьшит число "слепых" расследований и ускорит разбор проблем.

Для корреляции логов между сервисами необходимо присвоить сквозной уникальный идентификатор `trace_id` и передавать его в заголовке запроса при вызове
внутренних API "Александрит", также передавать его в поле сообщения в Rabbit. По аналогии передавать идентификатор заказа (`order_id`). В каждом
сообщении записанном в лог выводить `trace_id` и `order_id`.

Логи использовать совместно с метриками и трейсами:

* метрики позволяют обнаружить отклонение (рост 5xx, рост DLQ, увеличение latency);
* по алёрту инженер переходит к связанным логам по `service_name`, `order_id`, `message_id` и `trace_id`;
* при наличии `trace_id` расследование продолжается в системе трейсинга, где можно увидеть путь запроса между сервисами.

### Метрики, на которые повлияет логирование:

#### Технические:

* MTTR
* время диагностики инцидента
* количество инцидентов без установленной причины
* время поиска проблемного сообщения / заказа
* доля ошибок, классифицированных автоматически

#### Бизнесовые:

* количество обращений в поддержку по потерянным заказам
* среднее время ответа поддержки клиенту
* количество заказов с "непонятным статусом"
* удовлетворённость B2C и API-партнёров
* потери из-за сорванных сроков обработки заказов

### В первую очередь настроить логирование и трейсинг

* RabbitMQ integration
* CRM API
* MES API

#### Почему:

* здесь максимальная вероятность потери сообщения и зависания заказа;
* именно эти узлы влияют на рассинхрон статусов и "непонятные" состояния заказа.

### Во вторую очередь настроить логирование и трейсинг

* Shop API

#### Почему:

* нужен для полного end-to-end анализа, но основные инциденты уже локализуются на backend-интеграциях.

### В третью очередь настроить логирование и трейсинг

* ingress / frontend audit / дополнительные инфраструктурные логи

#### Почему:

* дают дополнительный контекст, но не решают основной бизнес-болезни.

## Предлагаемое решение:

### Стек:

* Filebeat и Logstash для сбора и передачи логов
* OpenSearch для хранения
* OpenSearch Dashboards для поиска и дашбордов

### Что нужно доработать в сервисах

* перейти на структурное логирование (JSON)
* в каждый лог добавлять:
    * trace_id
    * span_id
    * request_id
    * order_id
    * message_id
    * service_name
    * env
    * log_level

### Безопасность:

* доступ к логам только по RBAC
* роли: DevOps, backend engineers, support engineers
* доступ к UI только из корпоративной сети / через VPN
* чувствительные данные не логируются в открытом виде
* аудит доступа к системе логов
* в логах запрещено хранить чувствительные данные, которые не нужны для расследования инцидентов. В лог попадают только те поля, которые нужны для
  диагностики, корреляции и аудита
* не логируются вообще:
    * пароли
    * access token / refresh token
    * содержимое заголовков Authorization
    * session id / cookie
    * приватные ключи и другие секреты
    * полные платёжные реквизиты
* логируются только в маскированном или сокращённом виде:
    * email
    * телефон
    * адрес
    * контактные данные клиента
    * персональные данные (PII), если они необходимы для расследования
    * бизнес-идентификаторы клиента

### Политика хранения:

* отдельный индекс на систему и окружение
    * shop-prod-*
    * crm-prod-*
    * mes-prod-*
    * rabbit-integration-prod-*
* срок хранения:
    * INFO/WARN/ERROR в prod — 30–90 дней
    * DEBUG — не хранить постоянно или хранить очень коротко, например 1–3 дня
* размер:
    * rollover при 20–50 GB на индекс
    * удаление старых индексов после retention window

### Алёртинг:

* рост ERROR-логов по сервису
* всплеск DLQ / retry логов
* отсутствие ожидаемых событий по заказам
* резкий рост 5xx
* увеличение времени обработки заказа
* всплеск входящих запросов

### Поиск аномалий:

* резкий скачок создания заказов
* резкий рост повторных обработок сообщений
* всплеск ошибок в одном сервисе
* большое число заказов без смены статуса за N минут/часов
