### Проблемные места:

1. Критичные архитектурные проблемы:
    * Ненадёжный поток обработки заказа между Shop / CRM / MES.
    * Асинхронная интеграция без достаточного контроля доставки и обработки.
    * Рассинхрон статусов заказа между системами.
    * Слабая наблюдаемость по жизненному циклу заказа.
    * Низкая производительность MES dashboard для операторов.
2. Инженерные проблемы:
    * Ручные E2E как бутылочное горлышко релиза.
    * Недостаток автоматизированных тестов на интеграции и контракт взаимодействия.
    * Отсутствие метрик, трассировки, алёртинга.
3. Бизнес-последствия:
    * Рост просрочек.
    * Потеря контрактов.
    * Недовольство B2C и API-клиентов.

### Вероятные корневые причины текущих проблем:

* Рост нагрузки после запуска API выявил слабые места асинхронной интеграции.
* Отсутствуют надёжные механизмы контроля доставки и повторной обработки сообщений.
* Жизненный цикл заказа распределён между несколькими системами без явного определения system of record для статусов заказа.
* MES использует медленный сценарий чтения для отображения новых заказов.
* Недостаточная observability не позволяет быстро выявлять застрявшие заказы и деградацию интеграций.

### Инициативы для устранения нежелательных ситуаций в порядке убывания приоритета:

1. **Стабилизация потока заказа end-to-end**

   Описать ownership статусов, ввести correlation/order ID, аудит прохождения заказа по этапам, проверить обработку сообщений, retries, DLQ,
   идемпотентность consumers.
2. **Наблюдаемость и SLO по заказам**

   Метрики по времени перехода между статусами, число заказов в каждом статусе, lag очередей, доля "застрявших" заказов, алёрты на заказы без
   движения.

3. **Ускорение MES dashboard**

   Индексы, оптимизация SQL, read model для списка новых заказов, возможно отдельный быстрый endpoint "новые заказы в работе".
4. **Исправление интеграции через события**

   Outbox/inbox pattern, идемпотентная обработка, дедупликация, гарантии повторной доставки, schema/versioning событий.
5. **Улучшение релизного контура**

   Автотесты smoke + критические E2E + интеграционные тесты на статусы заказа и обмен через очередь.
6. **Функциональное разделение B2C и B2B нагрузки**

   Хотя бы на уровне очередей, rate limits, отдельных consumer-потоков и отдельных SLA.
7. **Ревизия доменной модели заказа**

   Нужен ли единый order service / master data / статусный оркестратор.

Проблема носит не инфраструктурный, а архитектурный характер: простое масштабирование инстансов не устранит потери и зависания заказов без исправления
потока обработки и интеграций.

### Если бы у меня была возможность выполнить только три самых приоритетных пункта, то я выбрал бы:

1. Основным приоритетом является стабилизация workflow заказа, т.к. из-за этого компания несёт основные финансовые и репутационные потери, также это
   препятсвует росту бизнеса. Данная инициатива может включить в себя другие инициативы, но вывод об этом можно будет сделать после того, как будет
   найдена причина нарушения рабочего процесса заказа. Данная инициатива имеет наибольшее влияние на выручку и SLA.
2. Выстраивание observability и алёртинга системы. Это важно для быстрой возможности оценивать состояние системы и получения возможности реагировать
   на проблемы до получения негативной обратной связи от клиентов. Это позволит быстрее выявлять инциденты, сокращать время их обнаружения и реакции,
   а значит - уменьшить количество просрочек и жалоб со стороны клиентов. Данная инициатива снижает MTTR и повышает управляемость системы.
3. Исправление первого запроса в MES. Данная инициатива приведёт к увеличению скорости взятия заказов в работу операторами, что приведёт к увеличению
   выручки компании за счёт выполнения большего количества заказов.

### Через 6 месяцев система должна прийти к состоянию, где:

* У заказа есть единый идентификатор и прозрачная трассировка между Shop, CRM и MES.
* Статусы меняются по понятным правилам, с определённым system of record.
* События доставки статусов и создания заказов идут через надёжный event flow: retry, DLQ, идемпотентность, мониторинг.
* Есть dashboard по SLA заказов и алёрты на "зависшие" заказы.
* MES UI для операторов получает отдельную быструю read model для новых заказов.
* Релизный процесс ускорен за счёт smoke/integration/critical-path test automation.
* B2B API-нагрузка не "ломает" B2C-поток.
