### Мотивация

Система мониторинга и алёртинга позволит компании «Александрит» повысить надёжность обработки заказов и сократить количество инцидентов, напрямую
влияющих на бизнес.

На текущий момент компания сталкивается с потерей заказов, задержками в обработке и жалобами клиентов (как B2C, так и API-партнёров). Отсутствие
прозрачности в работе распределённой системы (Shop → CRM → MES через RabbitMQ) не позволяет быстро выявлять причины проблем и устранять их.

Внедрение мониторинга даст следующие преимущества:

* Снижение потерь заказов:
  Позволит отслеживать сбои в очередях и интеграциях (например, зависшие или потерянные сообщения в RabbitMQ).
* Ускорение реакции на инциденты:
  Алёртинг позволит оперативно реагировать на рост ошибок (HTTP 500), деградацию производительности и перегрузку сервисов.
* Повышение SLA для API-клиентов:
  Мониторинг latency и ошибок позволит контролировать качество API и удерживать партнёров.
* Оптимизация производительности MES и CRM:
  Метрики помогут выявить узкие места (например, медленные запросы или перегрузку CPU), влияющие на работу операторов.
* Прозрачность жизненного цикла заказа:
  Наблюдаемость (observability) позволит понимать, на каком этапе теряются или задерживаются заказы.
* Обоснование масштабирования:
  Метрики нагрузки (RPS, CPU, memory) позволят принимать решения о масштабировании инфраструктуры на основе данных, а не предположений.

Таким образом, мониторинг напрямую влияет на выручку компании, снижая количество потерянных заказов и повышая удовлетворённость клиентов.

## Выбор подхода к мониторингу

### API (MES, Shop, CRM): 4 золотых сигнала.

#### Обоснование:

Для API выбран подход "четыре золотых сигнала", так как он позволяет оценивать сервис с точки зрения клиента: насколько быстро сервис отвечает,
сколько запросов обрабатывает, какова доля ошибок и не приближается ли сервис к пределу своей пропускной способности. Для Shop, CRM и MES API это
основной сценарий наблюдаемости, потому что именно через API пользователи и интеграционные клиенты взаимодействуют с системой.
RED можно было бы использовать как альтернативу для request-driven сервисов, но "четыре золотых сигнала" выбран как более полный.

#### Latency:

* p95 / p99 response time (excluding background processing)

#### Traffic:

* Number of requests (RPS)
* Number of requests (RPS) per user (for detecting abusive clients)

#### Errors:

* Number of HTTP 5xx
* Error rate (5xx / total requests)
* Number of HTTP 4xx (client errors)

#### Saturation:

* CPU %
* Thread pool utilization
* Memory utilization

### DBs (Shop, MES) — USE

##### Обоснование:

Для баз данных выбран подход USE, так как он лучше подходит для инфраструктурных компонентов. Для БД критично понимать уровень использования
ресурсов (connections, memory, CPU), признаки насыщения (slow queries, locks, saturation pool) и ошибки (timeouts, query errors). Подход "четыре
золотых сигнала" больше ориентирован на внешнее поведение сервисов, а USE — на внутреннее состояние системного ресурса. Подход RED ориентирован на
показатели важные для конечных клиентов.

Utilization:

* Memory utilization
* Number of connections

Saturation:

* Slow queries
* Lock wait / active locks
* Connection saturation (% of used connections from max pool)
* CPU utilization
* Disk I/O latency

Errors:

* Query errors
* Connection errors / timeouts

#### 3D file storage

Capacity:

* Size of S3 storage

Saturation:

* Disk I/O latency

#### RabbitMQ

* Number of messages in queue (queue length)
* Number of messages in flight
* Message processing lag (time in queue)
* Messages published per second
* Messages consumed per second
* Number of dead-letter-exchange messages
* Acknowledgement rate
* Retry rate
* CPU utilization
* Memory utilization
* Disk usage (for durable queues)

#### Business metrics

* Number of orders per status
* Time between status transitions
* Number of stuck orders (no status change > N minutes)

### План действий:

1. Развернуть Prometheus для сбора метрик.
2. Настроить exporters:
    * application metrics для Shop / CRM / MES;
    * RabbitMQ exporter;
    * node / DB exporters.

3. Развернуть Grafana и подключить Prometheus как datasource.
4. Создать дашборды:
    * API дашборд.
    * DB дашборд.
    * RabbitMQ дашборд.
    * Дашборд бизнес-метрик.
5. Настроить Alertmanager и правила алёртинга.
6. Определить пороги saturation и регламент реакции.
7. Проверить работу мониторинга в dev/release и затем включить в prod.

### Пороговые значения насыщенности и реакция системы:

#### API:

* CPU > 75% 5 минут подряд → warning alert
* CPU > 90% 10 минут подряд → critical alert, запуск масштабирования инстансов
* p95 latency > 1s → warning alert
* p95 latency > 3s → critical alert
* thread pool utilization > 80% → warning
* thread pool utilization >95% → critical

#### DB:

* connections > 80% от max → warning
* connections > 95% → critical
* slow queries > 10 (порог подлежит уточнению по результатам наблюдения) в минуту → alert DBA/DevOps
* disk latency выше порога → анализ индексов, нагрузочного профиля

#### RabbitMQ:

* queue lag > N минут → critical
* DLQ > 0 для критичных очередей → immediate alert
* Количество сообщений в очереди растут без снижения 5 минут → alert + тикет

#### Что происходит:

* создаётся alert;
* уведомление уходит в канал команды;
* для critical — эскалация DBA/DevOps, возможен автоматический звонок ответственному инженеру;
* для API можно включать горизонтальное масштабирование;
* для некритичных функций можно включить деградацию, например ограничить тяжёлые запросы.
